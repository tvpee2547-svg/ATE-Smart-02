## 1. บทนำโครงการและวัตถุประสงค์ (Introduction & Project Goals)
**A.T.E Smart Repair System** ถูกพัฒนาขึ้นเพื่อใช้เป็นเครื่องมือหลักในการปรับปรุงกระบวนการจัดการปัญหาซ่อมบำรุงภายใน **โรงเรียนทหารช่างกล ขส.ทบ.** จากระบบเดิมที่เป็นกระดาษไปสู่ระบบดิจิทัลบนคลาวด์แบบรวมศูนย์ (Cloud-based Centralized Digital Platform) 

### 🎯 เป้าหมายเชิงลึกในการพัฒนา:
* **แก้ปัญหาข้อมูลสูญหายแบบถาวร (High Data Availability):** ย้ายจากการเก็บข้อมูลในคอมพิวเตอร์เครื่องเดียว ไปเก็บไว้บน **Supabase Cloud PostgreSQL** ทำให้หมดปัญหาข้อมูลสูญหายจากไฟดับ เครื่องพัง หรือฮาร์ดดิสก์เสียหาย
* **การระบุเบอร์ติดต่อกลับทันที (Instant Contact Field):** เพิ่มช่องกรอกเบอร์โทรศัพท์ของนักศึกษาผู้แจ้งซ่อมไว้ถัดจากรหัสนักศึกษา ทำให้ช่างซ่อมบำรุงสามารถคลิกปุ่มโทรออกเพื่อสอบถามอาการหรือพิกัดสถานที่ได้ทันทีโดยไม่ต้องเสียเวลาค้นหาประวัติ
* **ความรวดเร็วในการกระจายงาน (Instant Dispatching):** ทำงานควบคู่กับระบบ **Socket.io** เมื่อมีการบันทึกข้อมูลเข้าฐานข้อมูลคลาวด์เรียบร้อยแล้ว ระบบจะยิงสัญญาณแจ้งเตือนและอัปเดตหน้าจอของทุกคนที่เกี่ยวข้องทันทีในเวลาเสี้ยววินาที

## 2. เจาะลึกเทคโนโลยีที่ใช้พัฒนา (Advanced Technology Stack & Hybrid Database Architecture)

ระบบนี้ผสมผสานเทคโนโลยีสมัยใหม่ที่ตอบโจทย์ความสถียร ความปลอดภัย และประสิทธิภาพสูง:

```
                  ┌──────────────────────────────────────────────┐
                  │                 FRONTEND                     │
                  │  React (Single Page Application) - Vite      │
                  │  TailwindCSS (Premium Responsive Utility)   │
                  │  Lucide Icons + Google Noto Sans Thai Font   │
                  └──────────────────────┬───────────────────────┘
                                         │  (JSON API / WebSockets)
                                         ▼
                  ┌──────────────────────────────────────────────┐
                  │                 BACKEND                      │
                  │  Node.js (Chrome V8 JavaScript Engine)       │
                  │  ExpressJS (Robust Minimalist REST API)      │
                  │  Socket.io (WebSockets Live Event Server)   │
                  │  Multer (Disk-Storage Binary Upload Handler) │
                  └──────────────────────┬───────────────────────┘
                                         │  (Dynamic Client Route)
                                         ▼
                  ┌──────────────────────────────────────────────┐
                  │          HYBRID DATABASE ADAPTER             │
                  │  Active: Supabase Cloud (PostgreSQL)         │
                  │  Fallback: Local Document DB (db.json)       │
                  └──────────────────────────────────────────────┘
```

### 💡 การออกแบบสถาปัตยกรรมฐานข้อมูลแบบ Hybrid Fallback (เทคนิคระดับสูง):
เพื่อความยืดหยุ่นในการทำงานและการจัดแสดงผลงาน ระบบหลังบ้านได้รับการพัฒนาให้มีระบบ **Database Fallback**:
1. **เมื่อกำหนดคีย์เชื่อมต่อครบถ้วน (`SUPABASE_URL`, `SUPABASE_ANON_KEY`):** ระบบจะทำงานบนระบบคลาวด์แบบเต็มรูปแบบ เขียน-อ่านข้อมูลจริงจากฐานข้อมูล PostgreSQL ของ Supabase ทันที
2. **เมื่ออยู่ในโหมดทดสอบแบบไม่มีอินเทอร์เน็ต:** ระบบจะตรวจจับอัตโนมัติและสลับไปใช้งานไฟล์ฐานข้อมูลท้องถิ่น `db.json` ทันที ทำให้ระบบไม่ล่มและสามารถนำเสนอผลงานออฟไลน์ได้โดยไม่มีข้อผิดพลาด!

## 4. โครงสร้างฐานข้อมูลคลาวด์ Supabase โดยละเอียด (Supabase PostgreSQL Tables)

ตารางทั้งหมดได้รับการออกแบบให้อยู่บนระบบฐานข้อมูลเชิงสัมพันธ์ (Relational Database) ของ PostgreSQL:

### 4.1 ตารางบัญชีผู้ใช้งาน (`users`)
ใช้เก็บข้อมูลสิทธิ์ สมาชิก และรหัสเข้าใช้งานระบบ
```sql
CREATE TABLE users (
  id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  username TEXT UNIQUE NOT NULL,      -- รหัสนักศึกษา 5 หลัก หรือชื่อเข้าใช้งานช่าง/แอดมิน
  password TEXT NOT NULL,             -- รหัสผ่าน ( Plain-text สำหรับทดสอบความเร็วในการนำเสนอ )
  name TEXT NOT NULL,                 -- ชื่อ-นามสกุลภาษาไทยของผู้ใช้งาน
  role TEXT NOT NULL CHECK (role IN ('student', 'employee', 'admin')), -- บทบาทผู้ใช้
  initials TEXT NOT NULL,             -- ตัวย่อโปรไฟล์ 2 ตัวแรก (เช่น 'สม', 'AD')
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 4.2 ตารางรายการแจ้งซ่อมบำรุง (`requests`)
ใช้เก็บข้อมูลปัญหาการแจ้งซ่อมทั้งหมด รวมถึงเบอร์โทรศัพท์และช่างผู้ดูแลรับผิดชอบ
```sql
CREATE TABLE requests (
  id TEXT PRIMARY KEY,                -- รหัสรายการใบงาน เช่น 'MR-5432'
  subject TEXT NOT NULL,              -- เรื่องที่แจ้งซ่อม (เช่น ท่อน้ำแตก, แอร์ไม่เย็น)
  category TEXT NOT NULL,             -- หมวดหมู่ (ไฟฟ้า, ประปา, แอร์, อาคารสถานที่, ไอที, ทั่วไป)
  reporter TEXT NOT NULL,             -- ชื่อจริงนักศึกษาผู้แจ้งซ่อม
  reporter_username TEXT NOT NULL,    -- รหัสนักศึกษา 5 หลักของผู้แจ้ง
  phone TEXT NOT NULL,                -- เบอร์โทรศัพท์ติดต่อกลับ (ตัวอักษร 9-10 หลัก)
  date TEXT NOT NULL,                 -- วันที่รายงานผลเป็นรูปแบบปฏิทินไทย (เช่น 22 พ.ค. 2569)
  status TEXT NOT NULL CHECK (status IN ('รอดำเนินการ', 'กำลังดำเนินการ', 'รออะไหล่', 'เสร็จสิ้น')),
  details TEXT,                       -- รายละเอียดอธิบายจุดเสีย พิกัดสถานที่ อาคาร เรียน
  assignee TEXT,                      -- ชื่อผู้ใช้ช่างซ่อมบำรุงที่รับผิดชอบเคสนี้
  image_url TEXT,                     -- ลิงก์ที่อยู่ของภาพหลักฐานปัญหา
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 📦 สคริปต์อพยพข้อมูล (`scratch/migrate.js`):
เป็นยูทิลิตี้ Node.js ที่ทีมพัฒนาสร้างขึ้นเพื่อดึงข้อมูลเก่าทั้งหมดในไฟล์ `db.json` ขึ้นระบบ Supabase โดยทำการเช็กซ้ำ (Duplicate Check) อัตโนมัติ ป้องกันข้อมูลประวัติการแจ้งซ่อมสูญหายระหว่างการเปลี่ยนผ่านระบบ

## 5. เจาะลึกการเขียนโค้ดส่วนสำคัญ (Code Walkthrough)

คำอธิบายตรรกะในส่วนโค้ดสำคัญ 3 ส่วนเพื่อช่วยในการอธิบายคณะกรรมการพรีเซนต์:

### 5.1 ระบบตรวจสอบสิทธิ์ในการดูข้อมูล (Role-Based Visibility)
ในหน้าแผงควบคุม มีการใช้ฟิลเตอร์คัดกรองข้อมูลเพื่อความปลอดภัยของข้อมูล โดยแยกเป็น 3 สิทธิ์:
* **นักศึกษา (student):** มองเห็นเฉพาะงานของตนเอง
* **ช่าง (employee) & แอดมิน (admin):** มองเห็นของนักศึกษาทุกคนเพื่อบริหารจัดการระบบ
```javascript
const visibleRequests = requests.filter((r) => {
  if (user.role === 'admin') return true;       // แอดมินดูได้หมด
  if (user.role === 'employee') return true;    // ช่างดูของทุกคนได้เพื่อเตรียมเข้าไปซ่อม
  if (user.role === 'student') return r.reporterUsername === user.username; // นศ.เห็นเฉพาะของตัวเอง
  return false;
});
```

### 5.2 การตรวจสอบความถูกต้องเบอร์โทรศัพท์ (Frontend Validation & Pattern)
ในหน้าแจ้งซ่อมใหม่ มีการบังคับให้อินพุตรับค่าได้เฉพาะตัวเลขผ่านฟังก์ชันอัปเดตสเตตัส และใช้นิพจน์ปกติ (Regular Expression) ตรวจสอบความถูกต้องของเบอร์โทรศัพท์ไทย (ต้องเริ่มต้นด้วยเลข 0 และมีจำนวน 9 ถึง 10 หลัก) ก่อนอนุญาตให้ยิง API บันทึกเข้าฐานข้อมูล:
```javascript
// บังคับให้อินพุตรับค่าได้เฉพาะตัวเลข 0-9
onChange={(e) => setPhone(e.target.value.replace(/[^0-9]/g, ''))}

// ตรวจสอบโครงสร้างเบอร์โทรศัพท์ก่อนกดบันทึก
if (!phone || !/^0[0-9]{8,9}$/.test(phone)) {
  alert('กรุณากรอกเบอร์โทรศัพท์ที่ถูกต้อง (9-10 หลัก เริ่มต้นด้วย 0)');
  return;
}
```

### 5.3 กลไกการเขียนข้อมูลลงคลาวด์แบบทำงานคู่ขนาน Socket.io
หลังบ้านจะทำการแทรกข้อมูลคำร้องแจ้งซ่อมลงตาราง `requests` ของ Supabase ผ่าน SDK จากนั้นจะสั่งให้ **Socket.io** ทำการอัปโหลดข้อมูลรายการอัปเดตใหม่ล่าสุดส่งกระจาย (Broadcast) ไปให้ทุกคนทันทีโดยผู้ใช้งานคนอื่นไม่ต้องกดรีเฟรชหน้าเบราว์เซอร์:
```javascript
// เขียนข้อมูลเข้าตาราง requests บน Supabase
const { error } = await supabase
  .from('requests')
  .insert({
    id,
    subject,
    category,
    reporter,
    reporter_username: reporterUsername,
    phone: reqPhone, // บันทึกเบอร์ติดต่อ
    date: formattedDate,
    status,
    details: reqDetails,
    assignee,
    image_url: imageUrl
  });

// โหลดข้อมูลชุดใหม่แล้วส่งสดให้ไคลเอนต์ทุกคนทันที
const updatedList = await getRequests();
io.emit('requests_updated', updatedList);
```

## 6. คู่มือการใช้งานสำหรับผู้ใช้งานแต่ละกลุ่มแบบก้าวต่อก้าว (Step-by-Step User Manual)

### 👨‍🎓 สำหรับนักศึกษา (Student Role)
* **ขั้นตอนที่ 1:** เข้าสู่ระบบด้วยรหัสนักศึกษา 5 หลัก
* **ขั้นตอนที่ 2:** กดปุ่ม **"แจ้งซ่อมใหม่"**
  * คุณจะพบช่องกรอกใหม่คือ **"เบอร์โทรศัพท์ติดต่อ"** แสดงอยู่ข้างๆ ช่องรหัสนักศึกษาอย่างเป็นระเบียบ
  * ระบุหัวข้อปัญหา เลือกหมวดหมู่ เขียนพิกัดจุดเสีย และแนบรูปหลักฐาน (ถ้ามี)
  * กดปุ่มส่งข้อมูล หากกรอกเบอร์โทรไม่ครบ 9-10 หลัก ระบบจะแจ้งคำเตือนทันที
* **ขั้นตอนที่ 3:** ติดตามสถานะงานซ่อมของตนเองผ่านการ์ดข้อมูลเรียลไทม์บนหน้าจอหลัก

### 👨‍🔧 สำหรับช่างซ่อมบำรุง (Employee Role)
* **ขั้นตอนที่ 1:** เข้าสู่ระบบด้วยบัญชีช่างซ่อมบำรุง (เช่น `somchai` / `1234`)
* **ขั้นตอนที่ 2:** ตรวจสอบตารางงานแจ้งซ่อมบนแผงควบคุมหลัก
